# pptx-go — product requirements from Deckard

> **Audience:** an agent (or contributor) working in *this* repo who wants to pick up the next
> engine work. This file states what **Deckard** — the agent-first slides product built on
> pptx-go — needs from the engine, and why, framed so you can implement a requirement without
> knowing Deckard's internals. Each requirement is a self-contained unit with engine-facing
> acceptance criteria.

## The relationship (read first)

pptx-go is an **engine, not a product** (D-026): it turns a typed `scene.Scene` into a
deterministic `.pptx` and nothing else. Deckard is the caller — it decides *what* slides to
emit, *what* text to write, *which* tokens to pick, and *whether* a layout warning matters.
**So these requirements ask for new engine MECHANISMS, never product opinions.** Deckard owns
the taste (palettes, when to use chrome, copy); the engine owns the faithful, deterministic
render of whatever typed scene it's handed.

The product's north star is an **agency-grade investor deck**: centered covers, dark full-bleed
section slides, cards with colored header bands + icon chips + status dots + watermark numbers, a
"VS" badge between compared cards, row-labeled bento grids, big-number stat/pricing strips, and
consistent chrome (a section eyebrow + rule, a footer logo + `02 / 11` page number). The engine
must be *capable* of all of that; Deckard drives it.

## Invariants every change MUST hold (the engine's contract)

1. **Deterministic.** Byte-identical output regardless of worker count. Pure functions, pinned
   integer-EMU constants, no `Date.now`/`rand`/map-iteration-order dependence. Add a worker-count
   determinism test (mirror `scene/render_parallel_test.go`).
2. **Additive + backward-compatible.** New capabilities are new optional fields whose **zero value
   reproduces today's render byte-for-byte**; the full existing scene suite must pass unchanged.
   (The ONE exception is R1 — see its note.)
3. **Unopinionated.** Provide the mechanism; do not bake in a palette, a default chrome, or a
   "make it pretty" heuristic. The caller supplies the values.
4. **Gate per unit:** `gofmt -l scene/` clean · `go build ./...` · `go vet ./...` ·
   `go test -race ./...` green, plus a golden/determinism test for the new behavior.

## What the engine ALREADY provides (do NOT rebuild)

- **Alignment** — `SceneSlide.Content = Alignment{Vertical VAlign; Horizontal HAlign}` + per-node
  `Align HAlign`; vertical top/center/bottom/justify of the body stack; horizontal via paragraph
  alignment for text (`scene/align.go`, `scene/render_leaves.go`).
- **Dark variant + full-bleed backgrounds** — `SceneSlide.Variant` (light/dark, dark derives a
  legible dark theme) and `SceneSlide.Background` (color/gradient/asset, full-bleed)
  (`scene/background.go`).
- **Deterministic text-width metrics** — `scene/metrics.go` (`naturalWidth`).
- **Rich card FIELDS** — `Card` already has `Eyebrow`, `Icon`, `HeaderPill`, `Fill`, `BorderStyle`,
  `Size`, `Layout` (icon-top), `Elevation`, and they render. (R4 only adds what's still missing.)

## Requirements (priority order)

### R1 — Content-aware text height  ·  HIGH  ·  ⚠ regenerates goldens

**Product need.** Agent-authored slides overlap and clip. `preferredHeight` (`scene/render.go`)
allots a fixed height per paragraph / per list item regardless of text length, so a paragraph
that wraps to 3 lines is given ~1 line of space and its text frame overruns into the next stacked
node. The same under-count means overflow is under-reported — content runs off the slide and the
renderer never warns, so the product can't tell the user "this slide is too full."

**Engine spec.** Make `preferredHeight` content-aware: estimate the wrapped line count from the
text runs, the resolved font metrics for the node's type role, and the available width
(reuse/extend `scene/metrics.go`), then `height = lines × line-height` (+ existing padding).
Apply to Prose (per paragraph), List (per item), Quote, Callout body, Table cells, Heading.
Keep it a pure, deterministic function (pinned chars-per-line / line-height constants).

**Acceptance.** A paragraph long enough to wrap to N lines is allotted ≥ N line-heights; the next
stacked node's Y is below the wrapped text's bottom (a golden asserts no overlap for a multi-line
fixture). A slide whose real wrapped content exceeds the body region emits the overflow
`LayoutWarning` (it currently doesn't). Determinism holds.

**Backward-compat note.** This is the ONE requirement that intentionally CHANGES layout for any
multi-line text, so existing golden snapshots must be **regenerated and eyeballed** (the new
output is the correct one — less overlap, truthful overflow). Single-line content is unaffected.

### R2 — Vertical fill / grow-to-fit  ·  HIGH

**Product need.** A heading + one block leaves the bottom half of the slide empty and reads thin.
Alignment's `VAlignCenter`/`VAlignJustify` *float* the block, but the reference look is the
**heading pinned at the top and the content GROWING to fill the frame** (tall cards, full bleed).

**Engine spec.** A fill mode (e.g. `VAlignFill`, or a per-node "flex" marker) where, after fixed
nodes (headings, prose) take their preferred height, the remaining body height is distributed to
the **flexible** nodes (containers: `Grid`, `TwoColumn`, `Card`, `CardSection`, `Table`; visuals:
`Chart`, `Image`) so they grow to consume it. Critically, the **container renderers must honor a
taller box** — `render_container.go` / `render_card.go` must distribute extra height to their
rows/cells/card bodies rather than rendering at a fixed height in a taller box.

**Acceptance.** A heading + grid slide renders the heading at top and the grid filling down to the
bottom margin; given a taller box, Grid/Card/TwoColumn produce proportionally taller cells. Zero
fill (today's mode) = byte-identical. Determinism holds.

### R3 — Slide chrome (section eyebrow + footer)  ·  MEDIUM

**Product need.** Reference decks read "designed" partly because every slide carries a section
eyebrow + hairline rule at the top (`01 — DIRECTION`) and a footer with a brand mark + a
`N / total` page number. The engine has no chrome concept, so Deckard can't produce it.

**Engine spec.** Optional, opt-in chrome rendered **outside** the body region (so it never
overlaps content): a top section-eyebrow band (eyebrow text + a hairline rule) and a bottom
footer band (a left brand slot — text or an asset id — and a right `N / total` page number).
Driven by fields on `SceneSlide`/`Scene` (e.g. `Scene` carries the brand slot + total; each
slide its section label + index). The body region shrinks to make room when chrome is enabled.

**Acceptance.** A deck with chrome enabled renders a consistent footer page number + section
eyebrow on each slide, outside the body box; chrome disabled = today's bare slide, byte-identical.

### R4 — Rich card visuals: header band, status dot, watermark  ·  MEDIUM

**Product need.** `Card` already supports fill/icon/eyebrow/headerPill/elevation. Reference cards
add three things it can't express: a **colored header band** (the top portion of the card a solid
accent color with the body in surface below — distinct from a full `Fill`), a **status dot** (a
small colored dot, top-right), and a **watermark** (a large faint number/label behind the card
content, e.g. a ghosted `01`).

**Engine spec.** Additive `Card` fields: `HeaderFill ColorRole` (a banded header region in that
color, body keeps `Fill`), `StatusDot ColorRole` (a small filled dot in the top-right corner),
`Watermark string` (a large, low-opacity label drawn behind the card body). Render in
`render_card.go`.

**Acceptance.** A Card with `HeaderFill` + `StatusDot` + `Watermark` renders all three (matches a
banded reference card); each zero-value omits its element; byte-identical when all unset.

### R5 — Composition primitives: center badge, inter-column connectors, row-labeled bento  ·  LOW

**Product need.** The reference compares two cards with a centered **"VS" badge** between them,
draws **connector arrows between 3 columns** (an architecture diagram), and uses a **row-labeled
bento grid** (rows with a left label and variable column spans).

**Engine spec.** (a) A `TwoColumn` option for a centered connector badge between the columns
(badge text). (b) Inter-column connectors for a multi-column container. (c) A grid/bento variant
with per-row left labels and variable column spans (`Ratio` per row already partly exists on
`Grid` — extend to row labels + spans). These can land as separate sub-units.

**Acceptance.** The two-card-with-center-badge slide and a row-labeled grid render; existing Grid/
TwoColumn unaffected when the new options are unset.

### R6 — Stat node (big number)  ·  LOW

**Product need.** Pricing and metric slides use big-number stats (`$2,200`, `38%`) with a label
and an optional delta. There's no engine node for a hero number, so they're faked with headings.

**Engine spec.** A `Stat` leaf node: `Value string` (rendered at display scale), `Label string`,
optional `Delta string` + `DeltaTone` (up/down/neutral color). A row of `Stat`s inside a `Grid`
forms a metric/pricing strip.

**Acceptance.** A `Stat` renders a large value + label (+ delta when set); a `Grid` of `Stat`s
renders a strip; validates in Stage-1.

### R7 — Expose resolved per-slide colors  ·  LOW (enables product-side contrast checks)

**Product need.** Deckard validates text/surface contrast, but it can't see the colors the engine
actually resolved per slide — especially for `VariantDark`, where the engine swaps to a derived
dark palette. The product validator currently checks contrast against the light theme and false-
flags white-on-dark.

**Engine spec.** Expose, per slide (e.g. in `Stats` or a query API), the **resolved** canvas /
surface / primary-text colors the engine rendered with — so a caller can compute true contrast
against the actual background, and apply large-text thresholds itself. No contrast *logic* in the
engine (that stays the caller's); just expose what was resolved.

**Acceptance.** After `Render`, a caller can read the resolved background + text colors for each
slide (including the dark-variant palette); documented; deterministic.

## How to pick up a unit

Pick one requirement, implement it under `scene/` (read `scene/render.go` + the relevant
`render_*.go`), add its golden + determinism test, run the gate, open a PR. Prefer the priority
order (R1 → R2 unblock the most product value and R1 unblocks truthful overflow). Keep every
change additive and deterministic per the invariants above. The caller (Deckard) will pick up each
capability by bumping its pptx-go dependency and wiring the new fields into its own contract +
guidance — you don't need to touch that side.

# Wave 2 — professional-bar requirements (R8–R14)

> Derived from a reference-vs-recreation gap analysis: a Deckard recreation of a professional
> investor deck was rendered and compared against the original, dimension by dimension. These are
> GENERALIZABLE capabilities (each handles a CLASS of pro decks, not one slide) needed to reach a
> professional bar for ANY input. Same invariants as R1–R7 apply: deterministic, additive,
> byte-identical when unused, unopinionated (engine = mechanism, caller = taste). 88 requirements.


## R8 — Brand-matched soul — palette, dark, multi-accent, fonts  ·  10 reqs (4 critical)

A deck must render in its OWN brand, acquired from a brand source (a .pptx kit and/or an NL brief) into a COMPLETE soul (every surface/text/semantic token + a soul-driven dark palette + brand gradients), established BEFORE building. The recreation rendered a branded deck in Deckard's default palette — the single biggest "template not designed" tell.


**R8.1 `soul-bootstrap-complete-palette`** — _CRITICAL · product_ — Bootstrap must seed a COMPLETE palette (every surface + text + semantic token), not 3 accents
- **Gap:** BootstrapParams (internal/soul/bootstrap.go) accepts only Accent/AccentAlt/AccentWarm + 3 fonts. It cannot set canvas, surface, surfaceAlt, success/error/info, or any of the 10 text roles. The reference brand is a coordinated system — warm cream surfaces (#F0ECE2 on ref-02/04/05), deep-navy canvas, jade/orange/violet/lime accents, AA-safe accent text — none of which a bootstrap call can fully express. The recreation therefore fell back to Deckard White surfaces (white/off-white) for every light slide and could only tint the one teal accent, so even where it tried (slide-05 orange card) the rest of the palette stayed default. P7 promises a COMPLETE soul (all tokens) but the bootstrap surface only exposes ~6 of ~24 color tokens.
- **Capability:** A single bootstrap call can specify (or omit, inheriting defaults) EVERY color token of a soul: all surface roles (canvas, surface, surfaceAlt, accent, accentAlt, accentWarm, success, warning, error, info), all text roles (primary, secondary, tertiary, inverse, muted, accent, accentAlt, success, warning, error), and the extension tokens (border, borderStrong, accentSoft). Any pro deck's full light palette can be captured in one structured object, with unset tokens inheriting the base soul deterministically.
- **Spec:** Replace the flat Accent/AccentAlt/AccentWarm fields on BootstrapParams with a complete optional Palette struct: Surfaces map[string]hex and Text map[string]hex keyed by the same token names Refine already validates (surfaceRole/textRole), plus an Extensions map[string]hex. Bootstrap clones DeckardWhite, then for each provided key parses the hex (reuse parseHexColor) and writes the resolved RGB into Theme.Colors.Surfaces/Text or Soul.Extensions; unset keys keep the inherited value byte-for-byte. Keep it additive: an empty Palette yields exactly DeckardWhite today. Validate unknown token names as an error (not silent drop). This is purely the seam between params and the existing Theme maps — no engine change.
- **Accept:** For any supplied light palette, a deck rendered through the bootstrapped soul resolves every slide's canvas/surface/card/accent/text color to the supplied hex (verifiable via the render stats.Colors hook in render_colors_test.go); omitting a token reproduces the DeckardWhite value byte-identically; an unknown token name returns a typed error.

**R8.2 `soul-from-brand-source`** — _CRITICAL · both_ — Acquire a brand-matched soul from a brand SOURCE (brand .pptx and/or natural-language brief) before building
- **Gap:** P7 claims a soul can be seeded 'from natural language and/or a brand .pptx', but Bootstrap only consumes hand-typed structured tokens — there is no path that reads a brand .pptx's theme1.xml/masters and emits a Soul, and no path that turns a NL brand brief ('deep navy, jade, orange, violet, lime accents, warm cream') into a palette. The recreation was built on the DEFAULT soul, which is exactly why the entire palette is wrong (every slide vs ref). The authoring flow never established a brand soul because nothing forced or enabled it from the brand source the reference clearly has.
- **Capability:** Given a brand artifact (a .pptx brand kit) OR a natural-language brand description, produce a COMPLETE, ready Soul (full light palette + dark palette + gradients + fonts) without the agent hand-enumerating 24 hexes. Extraction is deterministic and inspectable: the same brand .pptx always yields the same soul; the agent can then Refine specific tokens.
- **Spec:** Two additive ingestion paths feeding the same complete-Soul builder. (1) FromTemplate: read a brand .pptx via the existing pptx theme/template reader (load-a-brand-template skill / pptx.FromTemplate), map its theme color slots (dk1/lt1/dk2/lt2/accent1..6/hlink) and major/minor fonts onto the soul's surface/accent/text/font tokens by a fixed deterministic rule, and surface the result as a candidate soul for review. Accent1..6 populate the multi-accent palette (see soul-multi-accent-palette). (2) FromBrief: a structured-but-rich NL params object (named colors with roles, e.g. {role:'canvas', hex}, {role:'accent', hex}, plus 'mood:dark|light') that the agent fills from the brief; it calls the same complete bootstrap. Both must run at deck-creation time and the product MUST make 'no brand soul established' a loud, explicit state (see soul-establish-before-build).
- **Accept:** Feeding a brand .pptx whose theme defines navy/jade/orange/violet/lime yields a soul whose resolved palette matches those slots (token-by-token, deterministic across runs); feeding an equivalent NL brief yields a soul within a bounded delta of the same hexes; in both cases a 10-slide deck built on the result reads in-brand (no Deckard White cream/teal leaking) on every slide.

**R8.3 `soul-driven-dark-palette`** — _CRITICAL · both_ — Dark-variant palette must be SOUL-DRIVEN, not a pinned neutral gray
- **Gap:** _engine/scene/background.go hard-codes the dark variant to gray-900/800/700 (darkCanvas=#111827, darkSurface=#1F2937, darkSurfaceAlt=#374151, text F9FAFB/E5E7EB/9CA3AF) via darkThemeFrom, ignoring the soul entirely. The reference's dark slides (ref-03 The Shift, ref-07 White Label, ref-10 close, plus the ref-01/09 hero washes) are a RICH DEEP NAVY with subtle blue-violet depth — a brand color, not neutral gray. The recreation's dark slide-03 came out flat blue-gray (pinned), and slides 7/10 are a flat TEAL wash because the only way to escape the gray was to point a gradient at the accent role. Neither matches. The dark identity is unbrandable today.
- **Capability:** A soul carries its OWN dark palette (a full DarkSurfaces + DarkText token set, plus dark border/accentSoft extensions). darkThemeFrom uses the soul's dark palette when present and falls back to the current pinned gray only when the soul defines none — so any pro deck can render a brand-correct dark side (deep navy, ink, charcoal, etc.) deterministically, and a soul that defines no dark palette is byte-identical to today.
- **Spec:** Add an optional DarkPalette to the soul (DarkSurfaces map[ColorRole]RGB, DarkText map[TextColorRole]RGB, DarkExtensions map[string]hex). In darkThemeFrom(base, soul): if the soul supplies a dark palette, clone base and overlay the soul's dark surfaces/text/extensions; else keep the existing pinned darkCanvas/darkSurface/... constants verbatim (default unchanged). The soul package owns the dark palette field; the engine takes it as an explicit argument so determinism/byte-identity hold. Bootstrap/FromBrand seed a sensible dark palette (e.g. darken canvas toward the brand's deepest hue, not gray) but always overridable via Refine on a new 'darkSurface'/'darkText' category.
- **Accept:** A soul with a navy dark palette renders every VariantDark slide's canvas/surface/text to the soul's dark hexes (checkable via stats.Colors on a VariantDark slide); a soul with NO dark palette reproduces the current gray dark output byte-for-byte; the dark canvas is never a hue the soul did not specify.

**R8.4 `soul-multi-accent-palette`** — _HIGH · both_ — Support a multi-accent brand palette (4+ coordinated accent hues), not 3 fixed accent roles
- **Gap:** The reference uses FOUR-PLUS brand accents simultaneously and assigns them by meaning: jade/teal (primary), deep teal, orange (ref-05 'Way 03' card, slide-05 orange in recreation), violet/purple (ref-05 white-label '02' card, ref-07 'models' inline word), and lime/chartreuse (ref-07 'brand' word, ref-09/10 CTA pill). The theme exposes only ColorAccent/AccentAlt/AccentWarm. The recreation collapsed the palette to teal + a single orange — it lost violet and lime entirely (no purple card on slide-05, no lime CTA on slide-09/10), so the deck reads two-tone where the reference reads as a confident multi-hue system.
- **Capability:** A soul defines an ordered brand-accent palette of N coordinated hues (each with its own AA-safe on-light and on-dark text color), and scene nodes can select an accent by stable index/name (card 1 = accent[0], card 2 = accent[1], ...). Any pro deck that rotates 4-6 brand colors across cards/columns/sections renders each in its true hue; a soul that defines only the default 3 is unchanged.
- **Spec:** Add BrandAccents []AccentSpec to the soul (AccentSpec = {Surface RGB, OnLight RGB, OnDark RGB, Name string}); accent[0..2] default to the existing Accent/AccentAlt/AccentWarm so nothing changes when unused. Expose accent selection on the accent-bearing scene nodes (Card/Column/Bento/Stat headerFill, eyebrow, statusDot) via an additive optional AccentIndex (or named accent) field that resolves through the soul's BrandAccents; unset = current behavior. Bootstrap/FromBrand populate BrandAccents from brand accent1..6. Determinism preserved: index→hex is a pure lookup.
- **Accept:** A soul with [jade, orange, violet, lime] renders a 4-card row in those four hues with correct on-surface text contrast on each; reducing the palette to one accent renders all cards in that accent; a soul that sets no BrandAccents produces byte-identical output to today's 3-accent themes.

**R8.5 `soul-brand-gradients`** — _HIGH · both_ — Brand gradients (incl. a dark hero gradient) as named soul tokens, not ad-hoc role pairs
- **Gap:** BackgroundGradient (background.go/render.go) takes two raw ColorRole stops resolved through the active theme. There is no brand-defined gradient, and on a dark slide the stops resolve through the pinned gray dark theme — so the reference's signature deep-navy radial hero wash (ref-01, ref-07, ref-09, ref-10) is unreachable. The recreation's attempt (slides 7/10) had to aim the gradient at the accent role, yielding a flat teal wash with none of the navy depth or directional falloff the reference shows. The brand's hero gradient cannot be named, stored on the soul, or reused.
- **Capability:** A soul stores named gradient definitions (e.g. 'hero', 'heroDark') as explicit stop lists with angle/type, including stops drawn from the soul's dark palette. A slide can request a soul gradient by name and get the brand's exact wash, deterministically, on light OR dark variants. Souls that define no gradients are byte-identical to today's two-role gradient path.
- **Spec:** Add Gradients map[string]GradientSpec to the soul (GradientSpec = {Stops []{Pos float, Color RGB|ColorRole}, AngleDeg float, Radial bool}). Add a scene Background kind/field 'GradientName string' that, when set, resolves the named gradient from the active soul (light or dark palette as appropriate) and feeds pptx.LinearGradient/RadialGradient. Keep the existing two-ColorRole BackgroundGradient untouched for back-compat. RGB stops let a soul pin exact brand hues independent of variant; ColorRole stops still follow the variant. All values are caller-supplied — engine provides the mechanism only.
- **Accept:** A soul defining a 'heroDark' navy radial gradient renders identical navy washes on every slide that names it, on dark variants, across worker counts (byte-identical); decks that use only the legacy two-role gradient or no gradient are unchanged byte-for-byte.

**R8.6 `soul-contrast-aware-accent-text`** — _MED · both_ — Contrast-aware accent/on-surface text derivation, applied per variant
- **Gap:** derivedTextAccent (refine.go) just scales every channel by 0.78 — a crude darken that is neither perceptual nor contrast-checked, and it is only computed for the LIGHT theme. The dark variant (darkThemeFrom) never re-derives an accent text color, so an accent word on a dark navy slide keeps a light-theme accent hue and may fail contrast. The reference uses distinctly tuned accent text per background (legible jade on navy in ref-03/07, deeper jade on cream in ref-02). The recreation's inline accent emphasis is largely absent/flat (e.g. titles that should have one colored word render mono-color).
- **Capability:** For any accent surface and any background (light surface, cream, navy, dark), the soul can resolve an accent TEXT color that meets a target contrast ratio, computed per variant. Any brand color used as an emphasis word stays legible and on-brand on whatever surface it sits on, deterministically.
- **Spec:** Replace the 0.78 channel scale with a deterministic contrast-aware derivation: given accent RGB and a background RGB, compute WCAG relative luminance and nudge the accent toward darker/lighter until it clears a configurable target ratio (default ~4.5:1 for body, ~3:1 for large display), preserving hue. Compute TextAccent/TextAccentAlt for BOTH the light theme and the dark theme (so darkThemeFrom re-derives accent text against the dark canvas). Expose the derivation as a pure helper in the soul package so results are reproducible. Allow an explicit soul override to win over derivation.
- **Accept:** For a navy dark variant and a cream light variant, the derived accent text colors differ and each clears the target contrast ratio against its own background; an explicit override is honored verbatim; the derivation is a pure function (same inputs → same hex) so render output stays byte-identical.

**R8.7 `soul-dark-extensions-and-accents`** — _MED · both_ — Dark variant must re-resolve border/accentSoft/accent extensions, not inherit light-only values
- **Gap:** darkThemeFrom overrides only canvas/surface/surfaceAlt + 3 text roles; it leaves accent surfaces and ALL extension tokens (border #E0D5CA, borderStrong, accentSoft) at their LIGHT values. On a dark navy slide those warm-cream borders and light washes are wrong — they read as light-theme artifacts on a dark card. The reference's dark cards (ref-03, ref-07) use subtle dark-tuned hairlines and translucent fills, not cream borders. The recreation's dark cards inherit the light border/wash, contributing to the 'template' feel on dark slides.
- **Capability:** Every literal-stroke/wash extension token and accent has a dark-variant value that the dark theme uses, so dark cards get dark-appropriate borders, dividers, and accent washes. A soul that defines no dark extensions falls back to a deterministic dark default rather than the light value.
- **Spec:** Carry DarkExtensions in the soul (see soul-driven-dark-palette) and have the renderer read border/borderStrong/accentSoft from the active variant's extension set (the soul exposes ResolveExtension(token, variant)). When a soul provides no dark extension, derive a deterministic dark hairline (e.g. a low-alpha lightening of the dark surface) instead of using the light hex. Keep accent surfaces overridable per variant via the dark palette. Additive: souls without dark extensions and decks without dark slides are byte-identical.
- **Accept:** On a VariantDark slide, card borders/dividers/accent washes resolve to dark-variant values distinct from the light theme's cream tokens; a soul with no dark extensions yields a deterministic dark hairline (never the light cream hex); light-only decks are unchanged byte-for-byte.

**R8.8 `soul-establish-before-build`** — _CRITICAL · product_ — Authoring flow must establish (and confirm) a brand soul BEFORE building; default-soul use is a loud signal
- **Gap:** The recreation was built entirely on the DEFAULT Deckard White soul — the single biggest reason the colors are wrong vs every reference slide. Nothing in the flow required or even surfaced that no brand soul had been established before slides were authored. A pro deck has one brand identity decided up front; Deckard silently shipped 10 slides in the wrong palette.
- **Capability:** Deck creation makes the active soul explicit and first-class: building proceeds against a named soul, and falling back to the built-in default is an explicit, surfaced state (not a silent default) so an agent/human always knows whether the deck is in-brand. Works for any deck: brand-kit-driven, brief-driven, or intentionally-default.
- **Spec:** Make the deck carry an explicit SoulID and have the build/export path report which soul is active in its structured output and on the preview surface. When the active soul is the built-in default, emit a non-fatal UI-state/warning ('using built-in Deckard White — no brand soul established') in the tool result so the absence of a brand is visible, never silent. Provide the brand-acquisition tools (soul-from-brand-source) as the recommended first step and reference them in the warning. No new surface; reuse the existing Result + preview status pattern. This is product/contract wiring, not an engine change.
- **Accept:** Building or exporting a deck whose active soul is the default returns a visible 'no brand soul' signal in the structured result and on the preview; building on a bootstrapped/brand soul reports that soul's id/name and shows in-brand colors; the active soul id round-trips through export.

**R8.9 `soul-warm-neutral-surface-fidelity`** — _HIGH · product_ — Per-token surface fidelity: warm/branded neutrals (canvas, surface, surfaceAlt) must be soul-settable and reach the engine
- **Gap:** The reference's neutrals are themselves branded — a specific warm cream (#F0ECE2-ish on ref-02/04/05 cards) and a near-paper canvas, not generic white/gray. Because Bootstrap cannot set canvas/surface/surfaceAlt at all (only accents), the recreation's neutrals are whatever Deckard White ships; for a non-Deckard brand they would be wrong, and even matching brands can't tune them. Surfaces are ~60% of pixels on most slides, so a wrong neutral makes the whole deck read off-brand regardless of accent correctness.
- **Capability:** A soul can pin its exact neutral ramp (canvas, surface, surfaceAlt, border) to brand values, and those values reach every rendered surface (slide background, card fills, dividers) verbatim. Any pro deck's specific paper/cream/ink neutrals are reproducible, not approximated by generic white/gray.
- **Spec:** Folds into soul-bootstrap-complete-palette (canvas/surface/surfaceAlt are part of the complete Palette) but is called out as its own acceptance because neutrals dominate. Ensure the resolved soul Theme's neutral surfaces flow unchanged through the renderer for slide backgrounds, Card/Bento/Column fills, and border extensions; add a soul-fidelity test that a bootstrapped warm-cream soul produces those exact hexes on a representative multi-card slide.
- **Accept:** A soul pinning canvas/surface/surfaceAlt/border to specific brand neutrals renders a multi-card slide whose background, card fills, and dividers resolve to exactly those hexes (no generic white/gray substitution), verifiable via stats.Colors and shape-fill inspection.

**R8.10 `soul-roundtrip-verification`** — _MED · both_ — Soul→engine fidelity must be verifiable end-to-end (resolved render colors equal soul tokens)
- **Gap:** Because pptx-go code-authored themes do not round-trip through theme1.xml (noted in soul.go), the soul is re-applied per render — but there is no generalizable check that the FULL soul (light palette + dark palette + multi-accent + gradients + extensions) actually reaches the rendered bytes. **R14.16 `cov-soul-icon-set-and-duotone`** — _MED · both_ — Soul-level brand icon SET (registry) and duotone/two-tone icons, beyond per-render single-path extensions
